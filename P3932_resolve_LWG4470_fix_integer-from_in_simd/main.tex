\section{The issue(s) with \exposid{integer-from}}

\subsection{4470: The use of \exposid{integer-from}\code{<Bytes>} all over [simd] is incorrect for
[\ldots]}

In the discussion of \lwg{4238}, Tim noted that
\code{\exposid{integer-from}<Bytes>} does not work as intended because \code{Bytes} can be 16 after
\code{complex<double>} became a vectorizable type.\footnote{It was not intended to require
conforming implementations to now provide a 128-bit signed integer type.}
I then opened \lwg{4470}, which this paper aims to resolve.
\lwg{4470}:
\begin{quote}
  After the introduction of \code{complex<double>} to the set of vectorizable types, the
  \code{\exposid{integer-from}<Bytes>} trait does not work as intended anymore.
  All uses of \exposid{integer-from} need to be reviewed and the wording needs to be adjusted to
  work for \code{Bytes=16}.
\end{quote}

This issue is a question on how we define masks and their ABIs.
I sketched a potential solution in \url{https://lists.isocpp.org/lib/2025/04/31251.php}.

\subsection{4518: \code{simd::cat} return type requires inefficient ABI tag change / conversion}

Later I noticed that a surprising change in ABI can happen on \stdsimd\code{cat}.
This is \lwg{4518}:
\begin{quote}
  The return type of \code{simd::cat} is defined using \exposid{deduce-abi-t} rather than
  \code{resize_t}.
  This can lead to:

  \begin{lstlisting}
basic_vec<T, Abi> x = ...
auto [...vs] = simd::chunk<2>(x);
auto y = simd::cat(vs...);
static_assert(is_same_v<decltype(x), decltype(y)>); // can fail
  \end{lstlisting}

  This happens when bit-mask and vec-mask types are mixed, and can also happen (in my
  implementation) with complex value types.
  \code{simd::cat} should try to be conservative wrt.\ the resulting ABI tag.
  However, \code{simd::cat} allows different ABI tags on its arguments (to allow different width).
  If the user gives mixed input, there's no obvious correct answer.
  I suggest the simple heuristic of using the first argument with \code{resize_t}.
\end{quote}
Since the \code{simd::cat} wording currently uses \exposid{integer-from} it makes sense to resolve
the two issues together.

\subsection{4414: §[simd.expos.abi] \exposid{deduce-abi-t} is underspecified and incorrectly
referenced from \code{rebind} and \code{resize}}

Finally, since \lwg{4414} changes the same wording we need to change for \lwg{4470}, I incorporated
the proposed resolution into the wording of this paper.
\lwg{4414}:
\begin{quote}
  In 29.10.2.2 [simd.expos.abi], \exposid{deduce-abi-t} is specified to be defined for some arguments.
  For all remaining arguments, nothing is specified.
  This could be interpreted to make such specializations ill-formed.
  But that does not match the intent of making \code{simd::vec<std::string>} and
  \code{simd::vec<int, INT_MAX>} disabled specializations of \code{basic_vec}.
  (If \code{INT_MAX} is not supported by the implementation.)

  The wording needs to clarify what happens in those cases.

  In 29.10.4 [simd.traits], \code{rebind} and \code{resize} say "\exposid{deduce-abi-t}\code{<T,
  V::size()>} has a member type \code{type}".
  But that's not how \exposid{deduce-abi-t} is specified.
\end{quote}

\subsection{Modify [simd.expos.abi]}

\begin{wgText}[{[simd.expos.abi]}]\setcounter{Paras}{3}

\tcode{\exposid{deduce-abi-t}<T, N>} \wgChange{is defined}{names an ABI tag type if and only} if
\begin{itemize}

\subsection{Modify [simd.overview]}

\begin{wgText}[{[simd.overview]}]\setcounter{Paras}{0}
\pnum
Every specialization of \tcode{basic_vec} is a complete type.
The specialization of \tcode{basic_vec<T, Abi>} is
\begin{itemize}
 \item
    enabled, if \tcode{T} is a vectorizable type, and there exists value
    \tcode{N} in the range \crange{1}{64}, such that \tcode{Abi} is
    \tcode{\exposid{deduce-abi-t}<T, N>},
 \item
   otherwise, disabled, if \tcode{T} is not a vectorizable type,
 \item
   otherwise, it is \impldef{set of enabled \tcode{basic_vec<T, Abi>}
   specializations} if such a specialization is enabled.
\end{itemize}

If \tcode{basic_vec<T, Abi>} is disabled, then the specialization has a
deleted default constructor, deleted destructor, deleted copy constructor, and
deleted copy assignment.
In addition only the \tcode{value_type}, \tcode{abi_type}, and
\tcode{mask_type} members are present.

If \tcode{basic_vec<T, Abi>} is enabled, then
\begin{itemize}
  \wgItemAdd \tcode{basic_vec<T, Abi>} is trivially copyable,
  \wgItemAdd default-initialization of an object of such a type default-initializes all elements,
  \wgRem{and}
  \wgItemAdd value-initialization value-initializes all elements\iref{dcl.init.general}\wgAdd{,}
  \wgItemAdd[\tcode{basic_vec<T, Abi>::mask_type} is an alias for an enabled specialization of
  \code{basic_mask}, and]
  \wgItemAdd[\tcode{basic_vec<T, Abi>::size()} equals \tcode{basic_vec<T, Abi>::mask_type::size()}].
\end{itemize}

\end{wgText}
